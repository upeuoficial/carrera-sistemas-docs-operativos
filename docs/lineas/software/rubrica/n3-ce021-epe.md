# RÚBRICA NIVEL 3 — CE021 (Ingeniería de Requerimientos) — EPE

## CONTEXTO DE EVALUACIÓN

Se evalúa sobre:
- Definición completa del sistema antes de su construcción
- Incluye: SRS (CE0211), Prototipos (CE0212), Arquitectura (CE0213), UML (CE0214)

## PRINCIPIO

- NO evalúa si el sistema funciona ni si el código es bueno.
- Evalúa si el sistema está correctamente concebido y diseñado.

## CRITERIOS DE EVALUACIÓN

| Criterio                                 | Excelente (18–20)                                                                 | Bueno (15–17)                                   | Regular (13–14)                        | Deficiente (<13)                |
|------------------------------------------|-----------------------------------------------------------------------------------|-------------------------------------------------|-----------------------------------------|-------------------------------|
| Completitud y claridad de requerimientos | SRS completo, claro, estructurado; incluye RF, RNF, reglas de negocio y restricciones | SRS mayormente completo con omisiones menores   | Requerimientos incompletos o ambiguos   | Requerimientos deficientes o inexistentes |
| Coherencia entre requerimientos, prototipos y sistema | Prototipos y sistema reflejan fielmente los requerimientos definidos | Coherencia general con pequeñas inconsistencias | Varias inconsistencias                  | No hay coherencia              |
| Definición de arquitectura del sistema   | Arquitectura claramente definida (vistas, componentes, decisiones) y alineada al problema | Arquitectura funcional con limitaciones         | Arquitectura parcial o poco clara       | No hay arquitectura definida   |
| Modelado del sistema (UML)               | Diagramas UML completos (estructurales y de comportamiento) coherentes con arquitectura | Modelado adecuado con omisiones menores         | Modelado incompleto                    | No hay modelado                |
| Trazabilidad de requerimientos           | Cada componente del sistema puede rastrearse a requerimientos definidos                | Trazabilidad parcial                           | Trazabilidad limitada                   | No hay trazabilidad            |
| Validación con stakeholders              | Evidencia clara de validación de requerimientos y prototipos con usuarios              | Validación parcial                             | Validación limitada                     | No hay validación              |

## ESTRUCTURA DE LA RÚBRICA

- SRS
- Coherencia
- Arquitectura
- UML
- Trazabilidad
- Validación

## VALIDACIÓN (CLAVE)

- ✔ Alineada a IR + ADS
- ✔ No mezcla desarrollo (CE023)
- ✔ No mezcla calidad (CE024)
- ✔ Evalúa diseño real del sistema
- ✔ Totalmente trazable a evidencias CE021*

### DIFERENCIA CLAVE

| CE021                | CE023                |
|----------------------|----------------------|
| Define el sistema    | Construye el sistema |
| Diseño               | Implementación       |
| Documento + modelo   | Código               |

**Frase clave:**

> “Un sistema bien diseñado es aquel cuya implementación es una consecuencia lógica de sus requerimientos y arquitectura.”
